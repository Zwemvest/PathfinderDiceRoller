  <action>
1. Create `src/lib/db/index.ts` with the Dexie v4 class-based schema exactly as specified in the research doc:
   - `AppDatabase extends Dexie` with `super('PathfinderDiceRoller')`
   - Version 1 stores: `characters` (++id, name, importedAt), `rollHistory` (++id, rolledAt), `settings` (++id, key), `heroPoints` (++id, characterId)
   - TypeScript interfaces for all four entity types: `Character`, `RollHistoryEntry`, `Settings`, `HeroPoint`
   - Export the singleton `db` instance and all interfaces
   - Character interface: `id?: number`, `importedAt: Date`, `name: string`, `raw: string` (raw JSON blob for Phase 2)
   - RollHistoryEntry interface: `id?: number`, `rolledAt: Date`, `expression: string`, `result: number`
   - Settings interface: `id?: number`, `key: string`, `value: string`
   - HeroPoint interface: `id?: number`, `characterId: number`, `count: number`

2. Create `src/lib/db/persistence.ts`:
   - Export `requestPersistentStorage()` async function
   - Guard with `browser` import from `$app/environment`
   - Check if already requested via `db.settings.where('key').equals('persistenceRequested').first()`
   - If not, call `navigator.storage.persist()` and store the flag
   - Handle the case where `navigator.storage?.persist` doesn't exist (some browsers)

3. Create `src/lib/db/db.browser.test.ts` with browser tests:
   - Test: Dexie database opens successfully
   - Test: Can add and retrieve a settings entry (covers PLAT-05 round-trip)
   - Test: Can add and retrieve a character entry
   - Test: Data persists after closing and reopening the database
   - Use `beforeEach` to clear test data to avoid test pollution
   - Import `Dexie` and create a test-specific database instance (not the singleton) to avoid polluting app data

IMPORTANT: All Dexie operations in components MUST be guarded with `import { browser } from '$app/environment'`. The test file runs in browser mode so the guard is not needed there, but production code must have it.
